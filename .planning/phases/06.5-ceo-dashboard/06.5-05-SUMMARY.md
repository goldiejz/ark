## Live-refresh tick mechanism

```rust
loop {
    terminal.draw(|f| draw(f, app))?;
    if event::poll(Duration::from_millis(500))? {
        if let Event::Key(k) = event::read()? {
            if k.kind == KeyEventKind::Press { app.handle_key(k.code); }
        }
    }
    if app.last_tick.elapsed() >= Duration::from_secs(2) {
        app.refresh();
    }
    if app.should_quit { return Ok(()); }
}
```

Decoupling poll-timeout from tick-rate means key latency is sub-second (500ms worst-case) regardless of refresh cadence.

## Read-only delegation pattern (the `r` keybinding)

```rust
KeyCode::Char('r') => {
    if self.current_section == 1 {
        if let Some(id) = self.escalations_selected_id() {
            let _ = std::process::Command::new("ark")
                .args(["escalations", "--resolve", &id])
                .status();
        }
    }
    self.refresh();
}
```

The dashboard process **never opens ESCALATIONS.md for write**. It spawns the existing `ark escalations --resolve` command (the single-writer path established in Phase 4). After the spawn returns, the dashboard re-reads via `refresh()`. This honors REQ-DASH-04 (read-only) at the source-code level: `grep -E 'fs::write|OpenOptions::new\(\)\.write\(true\)' src/` returns zero hits in production code.

## Decisions Made

See frontmatter `key-decisions`. Highlights:

1. **Dual-mode `r` keybinding** — section-1-with-selection delegates to single-writer `ark escalations --resolve`; everywhere else just refreshes. Single key, two consistent semantics.
2. **Context-aware j/k** — section-3 scrolls rows, section-1 moves selection, others cycle. Preserves 06.5-04 muscle memory while adding scroll affordance.
3. **Greedy phase-prefix in drift** — `06.5-foo` extracts as `06.5` (not `06`) so `current_phase: 06.5-ceo-dashboard` matches against `06.5-foo` correctly.
4. **Help overlay over footer** — toggleable `?` overlay holds full keybinding doc; footer stays compact (one-line cheatsheet).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 2 - Missing functionality] `r` as universal refresh**
- **Found during:** Task 2 (plan said r-keybinding scope is "only fires on section 1")
- **Issue:** Plan described r as exclusive to section-1-resolve, but the user prompt asked for `r — refresh now (force tick)` as a global keybinding.
- **Fix:** Implemented dual semantics — section-1-with-selection delegates to single-writer; otherwise refreshes. Both behaviors call `self.refresh()` after the action. Documented in keybinding map.
- **Net:** plan invariant (single-writer delegation) preserved; user-prompt invariant (global refresh) added. Read-only audit still passes.

**2. [Rule 2 - Missing functionality] `?` help overlay**
- **Found during:** Task 2 (user prompt requested `?` toggle)
- **Issue:** Plan didn't enumerate help-overlay behavior; user prompt did.
- **Fix:** Added `app.show_help: bool` field, `?` toggles it, `draw_help_overlay()` renders centered Block+Paragraph. No new dep.

**3. [Rule 3 - Blocking] App.vault_path field**
- **Found during:** Task 2 (TierHealth needs vault_path to resolve verify-reports dir)
- **Issue:** App previously held only `portfolio_root`; tier_health needed `vault_path`.
- **Fix:** Added `vault_path: PathBuf` field to App, threaded through `App::new(db, portfolio_root, vault_path)`. main.rs constructs both from env vars.
