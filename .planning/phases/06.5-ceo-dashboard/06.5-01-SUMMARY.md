- **No TDD RED commit.** No separate test file was created — the plan's tests are inline in the verify block and run against the script itself. ## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] ANSI codes broke printf %-Ns column alignment in Section 4 (Recent decisions)**
- **Found during:** Initial real-vault test run
- **Issue:** `printf "%-22s ${color}%-18s${NC} ..."` counts the ANSI escape bytes against the field width, so the class column was inflated and the `[0m` reset code leaked into adjacent column whitespace. Visible as stray `[0m` characters in the rendered table.
- **Fix:** Pre-padded the class field with `printf "%-18s" "${class:0:18}"` into a local var, then wrapped that already-padded string in color codes. printf only sees `%s` for the colored portion, so column counting stays correct. Also split the colored vs no-color branch to avoid empty-color-string artifacts.
- **Files modified:** scripts/ark-dashboard.sh (Section 4 inner loop)
- **Verification:** Re-ran NO_COLOR=1 and color-on output; alignment correct in both modes; no leaked escape sequences.
- **Committed in:** 9943d90 (single atomic commit before this defect was visible — fix folded into the same commit since the script was not yet committed at that time)

### Schema flexibility (not a deviation, but worth noting)

- The plan's <interfaces> referenced `monthly_used_zar`/`monthly_cap_zar` keys for budget.json. The real vault uses `monthly_used` + `monthly_cap_tokens`. The python parser uses a `.get()` fallback chain to handle both, so the dashboard works on the actual data today and remains forward-compatible if the ZAR schema lands later. No request was made to change the source files — strict read-only.
